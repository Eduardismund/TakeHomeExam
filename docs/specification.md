# Family Budget Manager - Specification

## 1. Problem statement

The app helps a family (or any small group sharing money) keep track of who
earns what, who spends what, and where the money goes. Each person can be
added to the system together with their monthly income. Income and expense
categories are defined separately so the same category can be reused across
members. Whenever someone earns or spends money it is recorded as a
transaction tied to one member and one category.

At the end of the month the user can generate a report showing, for each
member, how much they earned, how much they spent, and the resulting balance.

It is a desktop application written in Java 17 with a Swing interface.
Data is kept in plain text files inside the `data/` folder so nothing is
lost between runs.

## 2. Entities

There are three entities, each with full create / read / update / delete.

### 2.1 Member

A person who participates in the budget.

| Field         | Type   | Notes                                |
|---------------|--------|--------------------------------------|
| id            | Long   | generated by the app                 |
| name          | String | the person's name                    |
| role          | String | free text, e.g. "Parent" or "Child"  |
| monthlyIncome | double | how much they normally earn / month  |

### 2.2 Category

A label that says what kind of money movement we're talking about.

| Field | Type         | Notes                          |
|-------|--------------|--------------------------------|
| id    | Long         | generated by the app           |
| name  | String       | e.g. "Salary", "Groceries"     |
| type  | CategoryType | either `INCOME` or `EXPENSE`   |

### 2.3 Transaction

One income or expense made by one member.

| Field       | Type      | Notes                                       |
|-------------|-----------|---------------------------------------------|
| id          | Long      | generated by the app                        |
| memberId    | Long      | id of the member that made the transaction  |
| categoryId  | Long      | id of the category it falls under           |
| amount      | double    | positive amount of money                    |
| date        | LocalDate | when it happened                            |
| description | String    | short note about the transaction            |

## 3. What the app must do

### 3.1 CRUD for each entity

For Members, Categories and Transactions the user can:

- add a new entry (the app refuses bad input - see section 4)
- list all the entries currently in the system
- update an entry by id (error if the id doesn't exist)
- delete an entry by id (error if the id doesn't exist)

Two extra rules around delete:

- when a Member is deleted, all their transactions are deleted too
- a Category cannot be deleted while transactions still use it

### 3.2 Functionality A: add a transaction (uses all three entities)

Recording a transaction is the part that ties everything together.
The user picks an existing member, an existing category, then types in the
amount, the date and a short description.

Before saving, the app checks that:

1. the chosen member exists
2. the chosen category exists
3. the values themselves pass validation (section 4.3)

If any of those fail, nothing is saved and the user gets an error message.
Otherwise a new Transaction is created with a fresh id.

### 3.3 Functionality B: monthly report

The user picks a month (in `YYYY-MM` form) and asks for a report.

For every member the app:

- finds all the transactions belonging to that member that happened inside
  the chosen month
- for each transaction, looks at the category type: if it's INCOME the amount
  goes to total income, otherwise to total expense
- computes balance = total income - total expense

The result is one row per member with `(id, name, total income, total expense,
balance)`. A member with no activity that month shows up with zeros.

## 4. Input rules

These are the rules used to decide whether an input is good or bad. They are
also what the black-box test cases (Task 2) will be built from.

### 4.1 Member

- `name`: must not be null, must not be empty / only spaces
- `role`: must not be null, must not be empty / only spaces
- `monthlyIncome`: must be zero or positive (a child with no income is fine,
  a negative number is not)

If something is wrong, a `ValidationException` is thrown that lists every
problem at once (so the user doesn't have to fix them one by one).

### 4.2 Category

- `name`: must not be null, must not be empty / only spaces
- `type`: must be one of `INCOME` or `EXPENSE`

### 4.3 Transaction

- `memberId`: must be a positive number
- `categoryId`: must be a positive number
- `amount`: must be strictly greater than 0 (zero is not a real transaction,
  negative makes no sense)
- `date`: required, and not in the future
- `description`: cannot be null, but is allowed to be empty

On top of those, when adding or updating a transaction the app also checks
that the `memberId` and `categoryId` actually point to existing entities in
the system.

## 5. Use cases

**UC-1 - add a member.** Open the Members tab, type the name, role and
income, press Add. The row shows up in the list.

**UC-2 - add a category.** Open the Categories tab, type the name, pick the
type from the dropdown, press Add.

**UC-3 - record a transaction (Functionality A).** Open the Transactions
tab, pick a member from the dropdown, pick a category from the dropdown,
type the amount, the date and a description, press Add Transaction.

**UC-4 - see the monthly report (Functionality B).** Open the Monthly Report
tab, type a month like `2026-05`, press Generate Report.

**UC-5 - fix a mistake.** Click on a row in any tab, change the values in
the form, press Update Selected.

**UC-6 - remove something.** Click on a row, press Delete Selected, confirm.

## 6. Other things worth noting

- **Architecture:** MVC. The code is split into `domain` (the entities),
  `repository` (loading and saving data), `validation` (the rules in
  section 4), `service` (the business logic that ties things together) and
  `ui` (the Swing screens).
- **Persistence:** three plain text files in `data/`, fields separated by
  semicolons.
- **Stack:** Java 17, Gradle, JUnit 5 and Mockito for tests.
- **Errors:** bad input throws `ValidationException`, missing ids throw
  `IllegalArgumentException`, the UI catches both and shows a popup.

## 7. Not in scope

- accounts and login - it is a single-user app
- multiple currencies or currency conversion
- recurring or scheduled transactions
- editing many things at once
- syncing with anything online
